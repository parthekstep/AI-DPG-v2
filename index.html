DPG Diffusion Architecture · v2.0

DPG Diffusion Architecture · v2.0 Proposal · June 2026

From a composable agent
to a composable ecosystem.

v1.0 let one team assemble one voice agent from 25 building blocks, configured in a single file, live in a day. The redesign lets an entire ecosystem — ministries, NGOs, startups — assemble many agents that share capabilities, talk to each other, and remember the citizen. The same building-block idea, moved up one level.

01

Where v1.0 got us

v1.0 took a real problem — every government voice deployment was a bespoke build — and solved it well. Twenty-five blocks across six tiers, wired through one YAML file, live in a day. A student scholarship helpline and a farmer advisory line run the same codebase, with only the configuration changing. That was the right first move, and it works today.

But every deployment is still an island. The composability stops at the edge of a single agent. The needs we now keep hearing from the field do not live inside one agent — they live in the space between agents.

02

Three stories from the fieldNone of these is a bug in v1.0. Each is a problem v1.0 was never shaped to solve.

Story A

The grievance flow, built three times

A scholarship helpline, a pension helpline, and a jobs line each need the same thing: take a complaint, file it on the right government portal, return a reference number, follow up. Three teams build it three times, to three different quality bars. When the grievance portal changes its API, three teams scramble — and the ones who don't, quietly break. A citizen who complains through one line gets a reference number and a follow-up; through another, a dead end.

The cost: duplicated effort, divergent quality, and a citizen experience that depends on which door they happened to knock on.

Story B

The citizen who falls through the gap

A construction worker calls the jobs line. Mid-conversation it emerges that he lost partial vision last year and is likely eligible for an assistive-device scheme and a maintenance allowance. The jobs agent knows nothing about disability schemes and has no way to bring in the agent that does. The best it can offer is another phone number — and the hope that he calls it, re-explains his whole situation to a stranger, and doesn't give up halfway. Most give up halfway.

The cost: the citizen carries the integration burden the system should carry. Every handoff is a re-explanation, and every re-explanation is a place to lose people.

Story C

The expert left at the door

An organisation like TRRAIN runs excellent interview prep and assessment for people entering retail jobs. When the jobs agent has profiled a candidate ready for exactly that, there is no clean way for that expertise to plug in — no way to publish it, no way for the agent to discover it, no way to offer the citizen a vetted choice, no way to know it isn't hallucinating. So the expertise stays outside, and the citizen gets a generic answer instead of the specialist who could actually help.

The cost: the ecosystem's best capabilities cannot reach the citizen, because there is no front door for them to come in through.

03

The same gap, in three disguises

v1.0 made blocks composable inside one agent. It has no model for composing across agents. That single missing idea shows up as four structural gaps.

  1. No shared capability layer. Common capabilities — grievance, eligibility, identity, document checks — are rebuilt in every deployment instead of built once and reused. Duplication is the default, and consistency is left to chance.
  2. No orchestration across agents. Agents are silos. A citizen must know which agent to call before they can get any help — the exact problem we set out to remove, reappearing one level up.
  3. No memory that travels with the citizen. Profile and history live inside a single deployment. Move to another agent and the citizen starts over from zero.
  4. No way for the ecosystem to contribute. Ministries, NGOs, and startups hold capability and knowledge the system needs, but there is no registry to publish it, no way to discover it, and no governance that lets others trust it.
The thread tying all four together: voice and chat are just the interface. The citizen should speak to one companion. Many agents and capabilities should work behind it, unseen.
04

The redesign: a composable ecosystem

The fix is not to discard the 25 blocks. It is to move the unit of composition up one level — from blocks inside an agent to agents and shared capabilities inside an ecosystem. The blocks become the substrate. Four roles sit on top.

Interface — the companionBuilt by the deploying partner
The one thing the citizen talks to — by voice or WhatsApp, in their language. It holds the conversation, knows the citizen, and coordinates everything else. It never exposes the machinery underneath.
Vertical agentsBuilt by the domain owner — KKB & DKB are ours
Purpose-built for one domain: jobs, pensions, water, scholarships. Deep and current in their subject and nothing else — which is exactly what keeps them reliable and easy to maintain.
Horizontal capabilitiesBuilt by whoever owns the capability — many are our DPGs
Reusable across every vertical: grievance filing, eligibility, identity, document verification, OCR, response testing. Built once, plugged in by anyone.
OrchestratorThe Manager Agent, promoted
Routes a citizen's need to the right vertical and pulls in the right capabilities — in parallel, holding one shared context so nothing is repeated. In v1.0 it routed within an agent; now it routes across them.

Two shared foundations turn this from a pile of code into an actual ecosystem:

The Capability RegistryThe marketplace
Where agents and capabilities are published, discovered, versioned, and governed. A ministry publishes its scheme agent; when the scheme changes, it updates once and every interface using it is instantly current. An NGO publishes a capability with a declared contract and published eval scores, so others can trust it before plugging it in. Nothing enters without passing a quality-and-safety gate.
Consented Citizen MemoryOwned by the citizen
A profile and history that belong to the citizen, not the deployment, and travel with them across agents — only ever with their explicit, in-the-moment consent. Tell the jobs agent your situation once; the schemes agent it hands you to already has the context you agreed to share.

For the builders in the room

Each capability is exposed through a standard interface (MCP or REST), published with a typed input/output contract, a version, and eval scores, and is independently deployable. The existing six tiers don't disappear — they describe what's inside a well-formed capability. And composition still happens in configuration, not code: v1.0 let you assemble an agent from blocks in one YAML; v2.0 lets you assemble a whole solution from agents and shared capabilities the same way.

One agent, assembled from blocks in one configMany agents and shared capabilities, assembled into one solution

Each agent a silo; the citizen picks the right doorOne interface; the orchestrator finds the door

06

Why this is the right shape

Separation of concerns, with one owner each

The interface keeps things simple. Domain agents stay accurate because the owner of the scheme maintains them. Capabilities stay reliable because each does exactly one thing. The registry keeps the whole thing trustworthy. Every layer has one job and one owner — the structure that has tripped up every monolithic citizen portal before it.

Build once, benefit everywhere

A capability improved in one place improves every deployment that uses it. The more that is published, the more useful the system becomes for everyone already in it. This is the diffusion in the name — finally true across the whole ecosystem, not just within one agent.

Resilient by construction

No single point of failure. If one domain agent is down, the others still serve. If a better grievance path opens, one capability updates and everyone has it — no portal redesign, no retraining, no outdated pamphlets still in circulation.

The citizen stays in charge

Memory belongs to them. Consent is asked at the moment of sharing, not buried in a terms screen. They can see, correct, and delete everything the system holds.

07

What we are not claiming

A redesign worth adopting should name its own limits. Four are worth stating plainly.

Voice has a latency budget

Every agent and capability added to a live call adds time. The orchestrator must keep the live path short — pull capabilities in parallel, and push anything that can wait to after the call. Subsecond chains of many agents are not free, and we will design against that ceiling, not pretend it away.

Routing quality becomes the make-or-break

When the orchestrator routes by reading each agent's description, that description — not the model — decides whether the citizen reaches the right place. Stale config and vague descriptions get executed faithfully, at scale. Routing discipline becomes something we own and test.

A shared registry is only as safe as its gate

Open capability registries elsewhere are full of unvetted, insecure entries. A public-good registry cannot inherit that posture. The quality-and-safety gate is not a feature to add later — it is the thing that makes the marketplace usable by a government at all.

It does not fix the back office

If the medical board behind a scheme is slow, this makes the wait visible and navigable — it does not make it shorter. We should be honest about what an interface can and cannot do.

The test that matters

A worker in Raichur speaks once, in Kannada, on a basic phone — and a jobs agent, a disability-scheme agent from another ministry, and three shared capabilities resolve his need together, without him knowing they were ever different systems, without repeating himself, and with the same quality he'd get anywhere else.

v1.0 made one agent easy to build. The redesign makes that sentence true.

Proving the model.
The smallest build that earns the bet.

The strategy proposal makes the case for a composable ecosystem. This is the build companion. It answers the questions that decide whether we start: what to build first, exactly which blocks, how voice fits, how the pieces talk, and what "proven" means. It is deliberately minimal — the goal of v1 is proof, not coverage.

01

What we are actually proving

The redesign rests on two claims. v1 exists to demonstrate both, end to end, in one working flow — not to ship a full marketplace.

Claim A · Reuse

A capability built once can be plugged into two unrelated agents and behave identically in both — with no rebuild, and a fix in one place reaching both.

Claim B · Orchestration

A citizen on one agent can be carried into another mid-conversation — context travelling with consent, one voice throughout — inside the voice latency budget.

One flow proves both: a worker on KKB (jobs) turns out to have a disability, and the companion pulls in a Schemes agent — both agents using the same Eligibility, Identity and Grievance capabilities. If that runs, the architecture holds.

04

How voice fits — and where the time goes

Voice is not an agent. It is the loop that wraps the whole system. The citizen speaks; the loop transcribes, routes, composes one reply, and speaks back. Everything in Section 02 happens inside that loop — which is exactly why latency is the question that decides the design.

The turn, step by step

Speech in→ASR (Bhashini)→Guardrails→Orchestrator route→Vertical agent→Horizontal capability (read)→LLM composes reply→TTS (Bhashini)→Speech out

Blue = on the live path, under the latency budget. Writes — filing the grievance, recording the outcome, running evals — happen after the call, off the live path.

Four rules that keep the live path fast

  • Co-locate everything that talks mid-call.

    Orchestrator, vertical agents and horizontal capabilities run in one process / one region. The manager→agent hop is then a function call, not a network round-trip — the difference between a few milliseconds and a few hundred.

  • Rules-first routing; LLM only for the unknown.

    Known intents ("file a complaint", "am I eligible") route by deterministic rules — effectively free. Only a genuinely novel ask spends one extra LLM call to decide where it goes.

  • Pull capabilities in parallel, not in sequence.

    When a turn needs eligibility and identity, fire both at once and wait for the slower one — not one after the other.

  • Mask depth with the voice itself.

    If a capability call will take time, the companion backchannels — "ठीक है, अभी देखता हूँ" — which buys real seconds while the work runs, the way a human agent does.

The question Tushar asked — how much does the extra hop add? Versus a single agent today, the only new step is the orchestrator's routing decision plus the manager↔agent call. Co-located + rules-first, a known intent adds ~0. A novel intent adds one LLM call (a few hundred ms), masked by a backchannel. Every tool/capability call is the same as it is today. Multi-agent does not multiply latency — it adds, at most, one maskable reasoning step.

One voice, always

Vertical agents and capabilities return structured data, never speech. Only the companion speaks. This is what stops two agents fighting over tone, turn-taking and persona inside one call — the failure mode that breaks naïve multi-agent voice. The machinery stays silent; the citizen hears one consistent companion.

05

How the pieces talk

Four contracts make composition possible. None is exotic — but all four must be fixed before code, or every team invents its own and reuse quietly dies.

  • Capability contract

    Every horizontal exposes a typed input → output over MCP or REST, with a version. A vertical calls it without knowing how it works inside.

  • Cross-agent handoff

    The orchestrator passes a consent-scoped context object — only the fields the citizen agreed to share — to the next agent. No silent leakage between ministries.

  • Consent ledger

    Every share is logged with what, to whom, and when. The citizen can see and revoke it. Consent is asked at the moment of sharing, not up front.

  • Registry entry + gate

    Nothing is callable until it is in the registry with a declared contract, version, and eval status — and has passed review. For v1 the gate is a manual review; that is enough to prove the discipline.

Shape of a capability entry (illustrative)

# registry-lite manifest — one entry per capability name: grievance version: 1.2.0 interface: mcp        # or rest input:  { user_token, issue_text, location, scheme_ref } output: { reference_no, portal, followup_at } consent_scope: [name, contact, location]   # fields it may receive evals: { groundedness: 0.94, task_completion: 0.91 } status: approved          # gate passed

Governance is not a separate product. Guardrails run in the live path on every turn; Evals run async on every agent and capability. Both already exist as blocks in v1.0 — here they simply become shared horizontals that everything passes through.

06

The path to proof

Six phases, each ending in a thing you can show. Reuse is proved before orchestration; voice hardening comes last, once there is something to make fast.

Phase 0

Contracts & skeleton

Fix the four contracts. Stand up the orchestrator skeleton and the registry-lite manifest.

Proof: a capability can be registered and called by a stub.
Phase 1

Build the three horizontals

Eligibility, Identity, Grievance — each standalone, contract-conformant, with eval scores.

Proof: each runs in isolation and passes its evals.
Phase 2

KKB consumes a horizontal

Rewire existing KKB to call the shared Grievance capability instead of its own logic.

Proof: Claim A, half — one vertical reusing a shared block.
Phase 3

Schemes vertical on the same blocks

Build the thin DEPwD agent; have it consume the same Eligibility + Grievance + Identity.

Proof: Claim A, full — two verticals, identical behaviour, zero rebuild.
Phase 4

Cross-agent handoff + memory

Orchestrator carries a consented context from KKB into Schemes mid-conversation.

Proof: Claim B — orchestration with consent, in text.
Phase 5

Voice end-to-end + hardening

Run the full flow over voice; apply the four latency rules; measure against the budget.

Proof: both claims, live, within the latency target.
07

Decisions to settle before code

Open questions that change the build. Each needs an owner and an answer before Phase 0 closes.

Where does consented memory live?

Do we host it centrally, or define a standard each interface implements and the citizen's chosen companion holds? This is a policy question as much as a technical one — and the answer affects trust with government partners. Pick a v1 stance even if it's provisional.

Owner · Product + Policy

Rules-first coverage target

What share of expected intents must route by deterministic rules (not LLM) to hit the latency budget? Set the number; it bounds how much routing logic Phase 0 writes.

Owner · Eng

The registry gate for v1

Is manual review acceptable to start, and who runs it? An automated security/eval gate is the eventual answer, but v1 needs a named reviewer, not a pipeline.

Owner · Platform

The Schemes slice

Confirm the exact DEPwD scheme set for the MVP. A tight slice — assistive devices + maintenance allowance — is demonstrable; "all disability schemes" is not.

Owner · Domain + DEPwD

The latency number

State the target out loud — e.g. perceived time-to-first-audio under a set threshold, with backchannel masking deeper calls. Without a number, "fast enough" is unfalsifiable.

Owner · Eng
08

What "proven" looks like

v1 succeeds when all of the following are true at once. Not before — and nothing else is required.

Definition of done

  • One horizontal capability is consumed by both KKB and the Schemes agent, with identical behaviour and no duplicated code.
  • A fix made to that capability once is reflected in both verticals without touching either.
  • A citizen on KKB is carried into the Schemes agent mid-call, with context travelling and every share logged in the consent ledger.
  • The whole flow runs over voice, within the agreed latency target, with one consistent companion voice throughout.
  • Every agent and capability in the flow has passing eval scores recorded in the registry.
